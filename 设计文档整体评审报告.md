## 🔴 开发前必须解决的阻塞项（8 项）

### 1. ~~多用户部署架构未明确~~ ✅ 已明确
- **来源**：需求文档 / 数据库设计
- **决策**：v1.0 明确为「单机单帐号」模式（系统仅内置唯一管理员帐号，拥有全部权限），SQLite 本地存储不通过网络共享。v2.0 计划迁移至 PostgreSQL 支持局域网多终端访问。
- **预留措施**：v1.0 开发中通过 Repository trait 抽象数据访问层、SQL 方言隔离在 adapter 模块、迁移脚本预留 PostgreSQL 版本，确保 v2.0 迁移成本可控。

### 2. ~~Next.js 16 版本风险~~ ✅ 已确认
- **来源**：需求文档 / 开发计划
- **决策**：确认使用 Next.js 16 正式版，版本风险已消除。

### 3. ✅ 生产管理模块边界缺失（已解决）
- **来源**：需求文档
- **决策**：v1.0 增加**最简化生产工单模块**，覆盖「创建工单 → 领料出库 → 完工入库」核心流程。工单表（`work_orders` + `work_order_materials`）已加入数据库设计，入库/出库类型增加 `production`，库存流水增加 `production_in`/`production_out`。UI 原型新增工单列表页和详情/执行页。排产计划、工序管理、报工等高级功能推迟至 v2.0。

### 4. ✅ 权限差异化 UI 方案缺失（已解决）
- **来源**：UI 原型设计
- **决策**：v1.0 调整为**单机单帐号**模式，系统仅内置唯一管理员帐号，拥有全部权限，不做角色差异和权限裁剪。权限差异化 UI（管理员/操作员视角差异）推迟至 v2.0 随 PostgreSQL 迁移和多帐号功能一同实现。v1.0 无需补充权限矩阵。

### 5. 时间戳使用 localtime 而非 UTC
- **来源**：数据库设计
- **问题**：所有 43 张表的 `created_at` / `updated_at` 均使用 `datetime('now','localtime')`。跨时区数据迁移（如越南工厂 → 中国总部查看）会导致时间混乱，且无法在事后修正历史数据。
- **建议**：统一使用 UTC（`datetime('now')`），前端展示时按 locale 转换。

### 6. `purchase_returns` 表零索引
- **来源**：数据库设计
- **问题**：该表包含 `supplier_id`、`inbound_id`、`status`、`return_date` 等高频查询字段，却无任何索引。随数据增长将严重影响采购退货查询和供应商对账性能。
- **建议**：补充 `supplier_id`、`inbound_id`、`status` 索引。

### 7. `reserved_qty` 缺少数据库约束
- **来源**：数据库设计
- **问题**：`inventory.reserved_qty` 和 `inventory_lots.qty_reserved` 均无 `CHECK(... >= 0)` 约束。若应用层 bug 导致预留数量为负，`available_qty` 生成列会产生不可信数据，影响库存预警和补货建议的正确性。
- **建议**：添加 `CHECK(reserved_qty >= 0)` 约束。

### 8. 开发计划缓冲时间严重不足
- **来源**：开发计划
- **问题**：23.5 周任务量仅 0.5–2.5 周弹性（4–10%），远低于行业建议的 15–20%。单人开发无法并行赶工，任何技术难点攻关或需求调整都会直接冲击交付日期。
- **建议**：将总工期调整为 26–28 周，在阶段三和阶段五各增加 1–2 周缓冲。

## 🟡 开发前建议明确的中等问题（18 项）

### 需求层面（6 项）

| # | 问题 | 说明 |
|---|------|------|
| M1 | 「其他出入库」缺乏审批管控 | 操作员可自由创建且立即生效，无限制地调整库存。建议增加金额/数量阈值限制 |
| M2 | 汇兑损益未考虑 | 采购单以 CNY 成交、实际付款时汇率变动产生的差异无处记录。建议增加「实际汇率」字段 |
| M3 | 销售单行折扣与整单折扣叠加规则不明确 | 明细行有折扣率，单头也有折扣率，两者如何叠加？建议明确：先行折 → 再整单折 |
| M4 | 负库存保护机制缺失 | 出库时「库存不足给出提示但不强制拦截」，可能导致负库存和成本计算异常。建议至少库存为零时强制拦截 |
| M5 | 智能补货触发时机不明确 | 「自动生成」是实时还是定时？建议明确为「用户进入补货页面时实时计算」 |
| M6 | 信用额度仅存储未使用 | 客户有信用额度字段但无拦截/警告逻辑。建议销售单审核时校验 |

### 数据库层面（5 项）

| # | 问题 | 说明 |
|---|------|------|
| M7 | `updated_at` 无自动更新机制 | SQLite 的 DEFAULT 仅作用于 INSERT，UPDATE 时不会自动更新。需应用层处理或创建 TRIGGER |
| M8 | `supplier_materials.currency` 缺少 CHECK 约束 | 与其他 currency 字段不一致，可能存入非法币种 |
| M9 | `supplier_materials` 缺少有效期字段 | 需求 §3.2.3.1 要求有效期起止，表中未体现 |
| M10 | 多表缺少日期/状态索引 | `stock_checks`、`transfers`、`sales_returns`、`payables`、`receivables` 缺少常用查询索引 |
| M11 | 迁移脚本幂等性和降级策略未具体化 | 仅文字提及，缺少实际写法示例 |

## 💡 总体评价

这是一套**质量高于行业平均水平**的设计文档体系。四份文档之间的**一致性良好**，核心业务逻辑（进销存闭环、多币种折算、批次追溯、成本核算双口径）的设计深度体现了对家具制造行业和进销存系统的深入理解。

**最大的亮点**：
- 金额处理全链路（整数存储 → 行级折算 → 头级汇总 → USD 基准）设计严谨
- 多币种/多语言从架构层面贯穿始终，而非后期补丁
- UI 原型在纯文本限制下做到了高信息密度和强一致性
- 开发计划的阶段递进和验收标准设计专业

**最大的风险**：
- v1.0 单机单帐号模式已明确，v2.0 PostgreSQL 迁移预留已做好
- 开发计划缓冲不足，单人开发容错空间小
- 缺少自动化测试覆盖，长期回归质量难以保障

**建议用 4–5 天完成阻塞项和中等问题的修补，即可正式启动 UI 设计和开发工作。**
