## 二、各文档详细评审

### 2.1 需求规格说明书 (01-requirements.md)

#### ✅ 亮点

- **越南本地化细致入微**：MST 税号、+84 电话、VND/CNY/USD 多币种、越南地址层级全部到位
- **金额舍入规则明确**：原币行级先舍入再汇总，USD 由 Rust 端统一计算，避免前后端差异
- **退货冲减模型优秀**：独立调整记录（`return_offset`）保证原单不可变，审计友好
- **成本双口径**：标准成本 vs 实际成本分离，出库时同时快照，完美支撑毛利分析
- **v2.0 迁移预留**：Repository 抽象、SQL 方言隔离、ID 策略预留，降低未来迁移成本
- **首次使用向导**：4 步引导流程设计简洁，降低上手门槛

#### 🔴 关键问题（3 项）

| # | 问题 | 位置 | 影响 | 建议修复 |
|---|------|------|------|---------|
| R-C1 | **销售折扣叠加公式重复扣折** | §3.5.1 第 725 行 vs 730 行 | 行金额已含 `(1 - 折扣率)`，汇总公式又乘了一次，导致应收金额计算错误 | 汇总公式改为 `Σ(行金额) × (1 - 整单折扣率) + 运费 + 其他费用` |
| R-C2 | **采购单头"应付金额"与实际应付生成口径混淆** | §3.4.1 第 595 行 vs §3.7.3 第 1017 行 | 采购单头的应付金额是订单预估，而 §3.7.3 的应付是按入库批次分摊生成的 | 将采购单头字段重命名为"订单总金额"或加注"预估" |

### 2.2 数据库设计 (02-database-design.md)

#### ✅ 亮点

- **金额 INTEGER 存储**：最小货币单位存储完全避免浮点精度问题，VND÷1/CNY÷100/USD÷100
- **多币种三层结构**：原币金额 + 汇率 + 基准币种(USD)金额，单据全链路可追溯
- **批次追踪三级模式**：`none/optional/required` 灵活适配不同物料
- **快照策略周密**：`unit_name_snapshot`、`*_by_name` 等审计快照保证历史数据不随主数据变更
- **预留两级结构**：`inventory_reservations`（物料汇总）+ `inventory_reservation_lots`（批次分配）
- **出库双成本口径**：标准成本快照 + 实际成本快照同时固化
- **GENERATED 列运用**：`available_qty`、`unpaid_amount` 等计算列减少应用层冗余计算
- **规约体系极其详尽**：金额折算、舍入规则、批次模型、预留策略等约定堪称模范
- **PRAGMA 配置专业**：WAL + busy_timeout + cache_size 配置合理

## 四、是否可以进入 UI 设计和开发阶段？

### ✅ 具备进入条件的理由

1. **四份文档高度对齐**：11 个功能模块、45 张数据表、30+ 页面全部有对应覆盖，跨文档一致性在同类项目中属上乘
2. **核心业务流程闭环**：采购→入库→库存+应付、销售→出库→库存+应收、退货→冲减，全链路定义完整
3. **技术方案可行**：Tauri 2 + Next.js + SQLite 的选型正确，中小工厂数据量下 SQLite 5 年内性能充裕（<200MB）
4. **8 项关键问题均为"可快速修复"类型**：公式修正、数字统一、约束补充，预计 2-4 小时可全部解决
5. **无架构级缺陷**：不存在需要推翻重来的设计问题

## 五、总结

### 🌟 设计文档体系的突出优势

1. **越南本地化深度罕见**：MST 税号、+84 电话、三语支持、多币种全链路，行业少有的细致
2. **金融精度处理专业**：整数金额存储 + 统一舍入规则 + Rust 端计算，从根源杜绝精度问题
3. **审计友好设计**：快照字段、退货冲减独立记录、操作日志，满足进销存系统的审计要求
4. **前瞻性架构**：Repository trait 抽象为 v2.0 PostgreSQL 迁移预留，成本双口径支撑多维分析
5. **文档间高度对齐**：需求→数据库→UI→开发计划形成完整闭环，这在实际项目中非常难得

### ⚡ 最需关注的 3 件事

1. 🔴 **立即修正销售折扣公式** — 这是唯一会直接导致财务数据错误的问题
2. 🔴 **统一跨文档数字** — 表数量、页面数量、工期的不一致会影响团队协作和预期管理
3. 🟡 **阶段三B 是最大风险点** — 定制单 + 生产工单 + 库存高级的组合复杂度高，建议增加缓冲

### 📌 最终建议

> **修复 8 项关键问题后即可启动开发。** 文档体系整体完成度高、质量优秀，完全具备支撑 UI 设计和开发工作的基础。剩余的重要问题和改进建议可在开发迭代中逐步完善，不构成阻塞。
