# 设计文档整体评审报告

## 1. 评审范围

- `docs/01-requirements.md`
- `docs/02-database-design.md`
- `docs/03-ui-prototype.md`
- `docs/04-development-plan.md`
- `README.md`

评审目标：判断现有文档是否足以支撑进入下一步的 UI 设计和 UI 开发阶段，并识别会直接影响设计、拆分开发和联调验收的关键缺口。

---

## 2. 总体结论

### 2.1 结论摘要

| 维度 | 结论 | 说明 |
|------|------|------|
| 业务需求完整度 | 较高 | 采购、销售、库存、BOM、定制、财务、补货、打印、多语言等主链路已基本覆盖 |
| 数据库设计成熟度 | 较高 | 核心表结构、批次、预留、多币种、日志、认证等主干已成形 |
| UI 设计准备度 | **可进入“深化设计阶段”** | 已有代表性原型，可继续补齐页面地图、组件规范、状态规范和高保真设计 |
| UI 开发准备度 | **暂不建议全面进入** | 存在若干跨文档阻塞项，尤其是财务闭环、单位换算追溯、页面覆盖不足 |

### 2.2 最终判断

- 可以进入 UI 设计阶段，但应定义为“补充与收敛阶段”，不是直接按当前文档进入最终视觉定稿。
- 不建议按当前文档直接全面进入 UI 开发阶段。
- 建议只先启动低风险工作：技术验证、项目脚手架、设计系统、布局骨架、登录认证壳、通用列表/表单组件。
- 在补齐本报告列出的 P0 阻塞项前，不建议同时推进完整业务页面开发和后端接口实现。

## 3. 文档的主要优点

- `docs/02-database-design.md` 对金额最小单位、USD 基准、数量精度、lot 模型、预留模型、单据单仓模式、会话安全都给出了明确约定，说明数据库设计不是停留在概念层。

- `docs/04-development-plan.md` 具备分阶段任务、验收标准和技术风险，整体上可作为项目排期底稿。

## 4. 关键阻塞项（P0）

### P0-1 财务闭环规则与数据结构未完全打通

**问题**：

- 需求文档定义了“采购入库生成应付、采购退货冲减应付”“销售出库生成应收、销售退货冲减应收”，见 `docs/01-requirements.md:512-513`、`docs/01-requirements.md:612`、`docs/01-requirements.md:623-624`、`docs/01-requirements.md:699`、`docs/01-requirements.md:848-874`。
- 但采购/销售单头才有折扣、运费、其他费用，见 `docs/01-requirements.md:538-543`、`docs/01-requirements.md:649-655`，数据库中对应字段也只存在于 `purchase_orders` / `sales_orders`，见 `docs/02-database-design.md:399-404`、`docs/02-database-design.md:567-573`。
- `inbound_orders` / `outbound_orders` 仅保留 `total_amount`，没有折扣、运费、其他费用分摊字段，见 `docs/02-database-design.md:462-463`、`docs/02-database-design.md:632-633`。
- `payables` / `receivables` 也没有与退货单的显式冲减关系或调整记录模型，见 `docs/02-database-design.md:967-1043`。

**影响**：

- 一张采购单或销售单分批执行时，单头费用如何分摊到每笔应付/应收没有可执行规则。
- 退货冲减应付/应收缺少数据闭环，后续财务页、对账单、报表和审计追溯都会出现口径不一致。
- 这不是纯 UI 问题，会直接阻塞接口设计、测试数据准备和验收口径。

**必须补齐**：

- 明确“应付/应收生成口径”到底按订单、按执行单，还是按执行单并做费用分摊。
- 明确退货是直接回写原应收/应付，还是生成独立调整记录。
- 为数据库补充必要的分摊字段或调整表。

### P0-2 单位换算与历史追溯模型不足

**问题**：

- 需求文档明确支持“基本单位 + 辅助单位 + 换算比例”，并要求业务单据按换算比例自动换算为基本单位入库/出库，见 `docs/01-requirements.md:225-229`、`docs/01-requirements.md:300-301`、`docs/01-requirements.md:325`。
- 数据库中物料主数据有 `base_unit_id`、`aux_unit_id`、`conversion_rate`，见 `docs/02-database-design.md:158-160`。
- 但采购、入库、退货、销售、出库等单据明细只保存 `unit_id + unit_name_snapshot + quantity`，没有保存“当时的换算比例快照”或“折算后的基本单位数量”，见 `docs/02-database-design.md:433-438`、`docs/02-database-design.md:486-489`、`docs/02-database-design.md:537-540`、`docs/02-database-design.md:603-609`、`docs/02-database-design.md:658-663`、`docs/02-database-design.md:713-716`。

**影响**：

- 一旦后续修改物料换算比例，历史单据数量、库存结转和成本回算可能失真。
- 这会直接影响库存、出入库、BOM、导入、报表和审计。

**必须补齐**：

- 在业务明细中保存 `conversion_rate_snapshot`。
- 明确是否要保存 `base_quantity` 作为库存与成本核算的唯一口径。
- 在需求、数据库、前端录单交互里统一这个规则。

### P0-3 “计量单位可自定义”已写入需求，但 UI 信息架构和开发计划缺失对应模块

**问题**：

- 需求文档明确写了“预置常用单位并支持自定义”，见 `docs/01-requirements.md:446-448`。
- UI 原型的左侧导航没有“单位管理”，见 `docs/03-ui-prototype.md:20-25`。
- 开发计划阶段二也没有“单位管理 CRUD 页面/任务”，见 `docs/04-development-plan.md:143-158`。

**影响**：

- 物料维护、数量精度、辅助单位换算、导入校验都依赖单位主数据。
- 缺少单位管理页意味着这个主数据只能靠种子数据硬编码，不符合需求。

**必须补齐**：

- 增加“单位管理”到信息架构、路由、原型和开发计划。
- 明确谁可以维护单位，修改单位后对历史数据是否只读追溯。

### P0-4 UI 原型覆盖仍不足以支撑“全面进入开发”

**问题**：

- 当前 UI 文档覆盖的是代表性页面，而不是全量业务页面，见 `docs/03-ui-prototype.md:91-987`。
- 需求与开发计划要求独立实现分类、供应商、客户、仓库、销售单、采购退货、销售退货等页面，见 `docs/01-requirements.md:330-445`、`docs/04-development-plan.md:146-152`、`docs/04-development-plan.md:189-200`。
- 现有原型中没有这些页面的完整列表/详情/编辑/状态流转原型。

**影响**：

- 开发会被迫边做边补交互，导致页面结构、筛选条件、字段分组、操作栏规则在开发中反复改动。
- QA 和验收也没有完整的页面级基线。

**必须补齐**：

- 补一份全量页面地图。
- 至少补齐 P0 模块的页面原型：分类、供应商、客户、仓库、销售单、采购退货、销售退货。
- 为关键页面补充“列表态 / 新建态 / 编辑态 / 审核后只读态 / 异常态 / 空态”。

## 5. 重要问题（P1）

### P1-1 库存金额口径前后不一致

**问题**：

- 需求文档在库存查询里写“库存金额 = 数量 × 参考进价”，见 `docs/01-requirements.md:728`。
- 但同一份需求文档又明确库存成本采用移动加权平均法，见 `docs/01-requirements.md:758-760`。
- 数据库库存表保存的是 `avg_cost`，见 `docs/02-database-design.md:740`。
- UI 原型展示了“可用库存总值 / 库存金额”，但没有说明口径，见 `docs/03-ui-prototype.md:379`、`docs/03-ui-prototype.md:392`。

**建议**：

- 明确库存金额到底按“参考进价”还是“移动平均成本”展示。
- Dashboard、库存页、报表页统一口径，避免同一系统里出现两套库存价值。

### P1-2 仓库“是否默认”字段与默认仓映射模型不一致

**问题**：

- 需求文档仓库字段里有 `是否默认`，见 `docs/01-requirements.md:437`。
- 同时系统设置又定义了“按原材料 / 半成品 / 成品维护默认仓映射”，见 `docs/01-requirements.md:935-939`。
- 数据库没有 `warehouses.is_default`，而是单独设计了 `default_warehouses`，见 `docs/02-database-design.md:275-289`。
- UI 原型和首次向导也都是按“默认仓映射”来表达，见 `docs/03-ui-prototype.md:649-654`、`docs/03-ui-prototype.md:1218-1222`。

**建议**：

- 删除仓库字段表中的“是否默认”，或重新定义其业务含义。
- 统一以“按物料类型映射默认仓”为唯一口径。

### P1-3 权限定义太粗，无法直接落成页面和按钮级开发

**问题**：

- 需求里只有管理员/操作员两级粗粒度说明，见 `docs/01-requirements.md:1043-1058`。
- 开发计划里已经要求做路由守卫、权限判断和用户管理，见 `docs/04-development-plan.md:112-114`、`docs/04-development-plan.md:131`、`docs/04-development-plan.md:157`。
- UI 原型也已经出现状态按钮与用户管理操作，但没有一份统一的页面/按钮权限矩阵，见 `docs/03-ui-prototype.md:299-302`、`docs/03-ui-prototype.md:608-610`。

**建议**：

- 增补一张“角色-页面-操作矩阵”。
- 至少明确导航可见性、列表按钮、详情按钮、审核/作废/导出/打印/恢复权限。

### P1-4 越南本地化场景提出了，但字段级校验规则还不够落地

**问题**：

- 需求文档前面写了越南电话、地址层级、MST 税号等场景，见 `docs/01-requirements.md:28-30`。
- 供应商、客户、企业信息页都出现了这些字段，见 `docs/01-requirements.md:377-385`、`docs/01-requirements.md:410-435`、`docs/01-requirements.md:943-945`。
- 但目前只有泛化的“支持表单校验”，缺少字段级规则和示例，见 `docs/01-requirements.md:1221`、`docs/03-ui-prototype.md:239`。

**建议**：

- 明确电话、邮箱、税号、地址字段的输入格式、长度、示例和校验错误文案。
- 明确不同国家/地区时字段是否动态变化。

### P1-5 视觉规范存在自相矛盾的描述

**问题**：

- UI 规范写“主色调搭配木色系”，见 `docs/03-ui-prototype.md:70`。
- 主题 token 里浅色主色是蓝色、深色主色是橙色，见 `docs/03-ui-prototype.md:1162`、`docs/03-ui-prototype.md:1177`。
- 字体规范写的是 `Inter + Noto Sans`，见 `docs/03-ui-prototype.md:73`。
- 开发计划的风险对策又写“跨平台使用系统字体栈”，见 `docs/04-development-plan.md:333`。

**建议**：

- 输出一份唯一的设计 token 基线。
- 一次性确定主色、语义色、字体栈、字号梯度、圆角、阴影、表格密度，不要在原型和计划里各写一套。

### P1-6 部分需求被放到较晚阶段，存在范围和排期上的优先级错位

**问题**：

- 三语完整翻译、深色主题、首次使用引导、操作日志查询页都已经出现在需求或 UI 原型中，见 `docs/01-requirements.md:186-199`、`docs/01-requirements.md:1060-1072`、`docs/01-requirements.md:1222`、`docs/03-ui-prototype.md:612-640`、`docs/03-ui-prototype.md:1196-1247`。
- 但开发计划把这些内容放在阶段五，且部分为 `P2`，见 `docs/04-development-plan.md:282-296`。

**建议**：

- 保留开发阶段划分可以，但要明确哪些只是“实现时间晚”，哪些属于“可降级到 v1.0.x”。
- 尤其是操作日志查询页和首次使用引导，建议不要等到最后再补。

## 7. 是否可以进入下一阶段

### 7.1 可以立即启动的工作

- 技术验证：`Next.js 16 + Tauri 2 + next-intl + SQLite` 的最小可运行样板，见 `docs/04-development-plan.md:96`。
- 设计系统：颜色、字体、间距、表格、表单、按钮、弹窗、状态标签、提示反馈等基础组件设计。
- 应用框架：左侧导航、顶栏、面包屑、认证壳、路由骨架。
- 低风险页面：登录、首页布局壳、物料管理基础页、系统设置中的汇率/编码规则/库存规则基础框架。

### 7.2 不建议直接全面启动的工作

- 财务模块完整开发
- 采购/销售分批执行后的应收应付联动开发
- 涉及辅助单位换算的库存与成本开发
- 全量业务页面并行开发
- 报表与看板的最终指标实现

## 8. 建议的准入清单

在进入“全面 UI 开发”前，建议至少补齐以下内容：

1. 输出《财务生成与退货冲减规则》一页文档，明确应收/应付生成口径、费用分摊规则、退货冲减模型。
2. 输出《单位与数量换算规则》一页文档，并同步修订数据库字段，补齐换算快照或基本单位数量字段。
3. 将“单位管理”补入信息架构、路由、原型和开发计划。
4. 补全 P0 页面原型：分类、供应商、客户、仓库、销售单、采购退货、销售退货。
5. 输出角色-页面-按钮权限矩阵。
6. 统一库存金额口径，明确库存页、Dashboard、报表页是否全部按移动平均成本展示。
7. 固化视觉 token 规范，消除主色和字体栈冲突。
8. 更新 `README.md` 的版本号、表数量和模块描述。

---

## 9. 建议的下一步顺序

1. 先补规则，不先补视觉。优先补财务闭环、单位换算、权限矩阵三件事。
2. 再补原型。把缺失的关键业务页面补成统一模板。
3. 再做设计系统收敛。统一 token、组件、交互状态。
4. 最后进入全面开发。此时再按 `docs/04-development-plan.md` 分阶段推进会稳很多。

## 10. 结论

现有文档已经具备“进入 UI 设计深化阶段”的基础，也具备启动脚手架和低风险前端骨架开发的条件；但还**不具备直接进入全面 UI 开发阶段**的条件。

根本原因不是“文档太少”，而是“几个关键口径还没有收口”：财务闭环、单位换算追溯、页面覆盖、权限矩阵、库存金额口径和设计 token 仍存在缺口或冲突。只要先把这些点补齐，整套文档就可以比较顺畅地进入设计定稿和分模块开发。
