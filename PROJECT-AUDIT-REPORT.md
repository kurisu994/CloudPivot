## 三、架构评估

### 3.1 整体架构 — ⭐⭐⭐⭐⭐ (5/5)

```
┌─────────────────────────────────────────────────────┐
│                   Tauri 2 Shell                       │
├──────────────────────┬──────────────────────────────┤
│   Next.js 16 (SSG)   │      Rust Backend            │
│   ┌──────────────┐   │   ┌────────────────────┐    │
│   │ React 19     │   │   │ commands/ (17 mod)  │    │
│   │ + shadcn/ui  │◄──┼──►│ + auth + db + error │    │
│   │ + next-intl  │IPC│   │ + inventory_ops     │    │
│   │ + Recharts   │   │   │ + operation_log     │    │
│   └──────────────┘   │   └────────┬───────────┘    │
│                       │            │                 │
│                       │   ┌────────▼───────────┐    │
│                       │   │ SQLite (WAL, 45表)  │    │
│                       │   └────────────────────┘    │
└──────────────────────┴──────────────────────────────┘
```

**优势：**
- Tauri 2 + SQLite 的桌面端选型精准，启动快、体积小、离线可用
- 前后端职责分离清晰：Rust 处理数据和业务逻辑，Next.js 纯 UI 渲染
- `isTauriEnv()` 兼容 Tauri 2 运行时识别，降级机制使前端可独立开发调试
- 自管理迁移框架 + `include_str!` 编译时嵌入，可执行文件自包含

### 3.2 后端架构 — ⭐⭐⭐⭐ (4/5)

**优势：**
- 统一 `AppError` 错误类型（5 种变体），所有命令返回 `Result<T, AppError>`
- 事务使用规范（begin → 操作 → commit），错误自动回滚
- 金额使用整数存储（最小货币单位），避免浮点精度问题
- `inventory_ops.rs` 抽取 10 个可复用函数，供采购/销售/盘点/调拨共用
- 操作日志系统完整，关键业务操作均有审计记录
- 编号生成在事务内执行，保证唯一性

**不足：**
- 所有写操作硬编码 `user_id=1`，多用户场景无法追溯操作人 → ✅ 已修复（CurrentUser state）
- 动态 SQL 构建存在两种风格（手动拼接 vs `QueryBuilder`），不统一
- count/data 查询条件重复（每个列表命令约 80 行重复代码）
- ~~仅 7 个单元测试（data_management: 3, reports: 4），核心业务逻辑无覆盖~~ → ✅ 已修复（36 个测试）
- 库存数量使用 f64 浮点类型，大量出入库后可能累积误差
- 采购/销售模块高度对称但完全复制（4,815 行 DRY 违规） → ✅ 已修复（提取 `order_shared.rs` 共享抽象）
- ~~数据库初始化失败直接 panic，无优雅降级~~ → ✅ 已修复（降级为内存数据库 + DbInitError 标记）

### 3.3 前端架构 — ⭐⭐⭐⭐ (4/5)

**优势：**
- 页面入口统一模式：`async function Page` + `setRequestLocale`
- 列表页统一使用 `BusinessListTableShell` 骨架组件
- 所有文案通过 `t()` 获取，无硬编码
- 状态管理纯 React hooks，无额外状态库（项目规模适配）
- 错误/成功反馈统一 `toast.error()` / `toast.success()`
- AuthProvider 路由守卫 + 钥匙串持久化 + session_version 校验

**不足：**
- ~~`lib/tauri.ts` 达 3,774 行，API 封装与 mock 数据混合，维护困难~~ → ✅ 已修复（拆分为 15 个子模块）
- ~~Dashboard 7 个数据面板的 `catch {}` 为空，错误被静默吞掉~~ → ✅ 已修复
- ~~缺少 Next.js 错误边界（`error.tsx`、`not-found.tsx`）~~ → ✅ 已修复
- Loading 状态不统一（骨架屏/旋转图标/纯文本/无状态混用）
- Dashboard 快捷操作按钮为纯占位符（无 onClick/href）
- 面包屑导航仅覆盖二级路径，深层页面回退到 "Dashboard"
- 缺少统一前端测试框架，现有轻量 Node 测试尚未接入 package.json test 脚本
- 无障碍缺失（0 处 `prefers-reduced-motion`）
